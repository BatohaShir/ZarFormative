## Решена ли исходная проблема (медленные переходы)?

### Да — проблема решена комплексно на 4 уровнях:

**1. Network layer (самый большой impact)**

- Middleware: `getUser()` убран с публичных маршрутов → -100-300ms на каждом переходе
- Это был корневой bottleneck — сетевой запрос к Supabase Auth на КАЖДОЙ навигации

**2. Perceived performance**

- `loading.tsx` skeletons → мгновенный визуальный отклик вместо "зависания"
- Пользователь видит skeleton за <50ms, даже если данные грузятся 500ms

**3. Bundle & rendering**

- SSR-конверсия 4 account pages → server component shell рендерится мгновенно
- Dynamic import тяжёлых компонентов (request-detail-modal, ElapsedTimeCounter) → меньше initial JS
- Map constants isolation → нет случайного Leaflet leak в SSR

**4. Assets**

- PNG → WebP → 47-68% экономии на иконках
- Bundle analyzer → инструмент для мониторинга регрессий

### Что проект имел изначально (уже было хорошо):

- ISR с revalidate (60s-1800s)
- Promise.all для параллельных запросов
- React Query с staleTime 5 мин
- Dynamic imports для модалок и карт
- Денормализованные поля (нет N+1)

### Ожидаемый суммарный эффект:

- Публичные страницы: **-100-300ms** на переход (middleware fix)
- Account pages: **мгновенный shell** + code-split client load
- Perceived performance: **<50ms** до первого visual feedback (loading.tsx)
- Bundle size: **уменьшен** за счёт code-splitting и dynamic imports
